## 报告核心升级：问题发现 → 根因分析 → 决策建议 → 自治愈

升级后的巡检报告不只在表层发现问题，而是驱动**完整闭环**：

```
┌─────────────────────────────────────────────────────────────┐
│  ① 问题发现 (Discovery)                                     │
│  └─ 巡检结果 → 定位异常指标/事件                            │
│                                                             │
│  ② 根因分析 (RCA)                                          │
│  └─ 下层联动 → 关联日志 → 时间线追溯 → 找到根本原因       │
│                                                             │
│  ③ 决策建议 (Decision)                                     │
│  ├─ [MANUAL]     人工决策     — 需人工判断后操作            │
│  ├─ [AI_ASSIST]  AI辅助决策   — AI推荐方案，人工确认后执行 │
│  └─ [AUTO_HEAL]  自治愈       — AI自动执行，事后通知        │
│                                                             │
│  ④ 响应时效 (SLA)                                           │
│  └─ P0(15min) > P1(2h) > P2(24h) > P3(下一迭代)            │
└─────────────────────────────────────────────────────────────┘
```

---

## 一句话触发

> **"帮我做一次全面运维巡检，包含根因分析和自治愈建议"**

Agent 执行三层全量巡检 → 逐项根因分析 → 按决策类型分层输出。

## 完整提示词模板

```
帮我做一次全面的运维巡检 + 根因分析，按以下三层自底向上执行：

1. 网络层：检查 ALB/NLB 健康状态、NAT Gateway 连接数/丢包、VPC 路由表配置、EIP 绑定状态、安全组规则
2. 云资源层：基于网络层结果，检查 EC2 实例状态/CPU/磁盘、RDS 连接数/存储/备份、ElastiCache Redis 命中率/内存、
   DocumentDB 副本延迟
3. 应用层：关联下层结果，深入 EKS 集群检查 Node/Pod 状态、CoreDNS 解析、Ingress/Service 连通性

对每一个发现的异常，按以下格式输出完整的发现-分析-决策-闭环：

【发现】异常资源名 + 当前状态 + 偏离标准的差值
【RCA】根因分析：
   ├─ 直接原因（当前层）
   ├─ 关联下层异常（如果有）
   └─ 判定依据（引用日志/指标/配置差异）
【决策】建议的操作方案
   ├─ 决策类型：[MANUAL] / [AI_ASSIST] / [AUTO_HEAL]
   ├─ 预期效果：操作后恢复什么
   └─ 失败回退：如果操作失败，下步怎么办
【SLA】响应时效：P0(15min) / P1(2h) / P2(24h) / P3(下一迭代)

最后输出汇总表和一个可执行的自治愈动作列表。
```

## 决策类型定义

| 类型 | 标签 | 含义 | 响应时间 | 示例场景 |
|------|------|------|----------|----------|
| 人工决策 | `[MANUAL]` | AI 识别问题 + 定位根因，但操作需人工判断 | > 4h | 安全组规则变更、升降配审批、删除资源 |
| AI 辅助决策 | `[AI_ASSIST]` | AI 给出推荐方案 + 预期效果 + 成本，等待人工确认 | 1-4h | 实例规格调整、存储扩容、集群扩缩容 |
| 自治愈 | `[AUTO_HEAL]` | AI 自动执行修复，执行完毕后通知用户 | < 15min | Pod OOM 资源调整、ASG 扩容触发、无状态服务重启 |

### 自治愈的边界条件

自治愈并非无条件执行。以下情况 **必须降级为 AI_ASSIST 或 MANUAL**：

| 条件 | 降级目标 | 原因 |
|------|----------|------|
| 涉及数据删除 | `[MANUAL]` | 数据安全不可逆 |
| 跨账号操作 | `[MANUAL]` | 需跨账号授权 |
| 成本变更 > $100/月 | `[AI_ASSIST]` | 成本影响需用户知晓 |
| 第一次出现的异常类型 | `[AI_ASSIST]` | 无历史模式可供参考 |
| 自治愈失败 2 次后 | `[MANUAL]` | 防止反复失败造成雪崩 |

## 裁剪版本（含根因分析）

### 仅网络层
```
帮我巡检网络层并做根因分析：
检查 ALB/NLB 目标组健康状态、NAT Gateway 丢包情况、
VPC 路由表、EIP 绑定、安全组规则。
对发现的每一个异常，写明 RCA 和决策建议。
```

### 仅应用层
```
帮我巡检 EKS 集群并做根因分析：
检查 Node/Pod 状态、CoreDNS、Ingress、节点压力。
对 CrashLoopBackOff 或 Pending 的 Pod，做自动根因诊断。
```

## 触发方式总结

| 触发词 | 执行范围 | 决策类型覆盖 |
|--------|----------|-------------|
| "帮我做全面巡检+根因分析" | 三层全量 | AUTO_HEAL / AI_ASSIST / MANUAL |
| "帮我巡检网络层并做 RCA" | 仅网络层 | 同全量 |
| "帮我看看 EKS 为什么有问题" | 仅应用层 + 根因诊断 | AI_ASSIST 为主 |
| "帮我排查一下 EC2 CPU 高的原因" | 单资源 + 深度 RCA | AUTO_HEAL (ASG) / AI_ASSIST (扩容) |
| "自治愈巡检" | 三层全量 | 仅 AUTO_HEAL 项执行，其余列出等待确认 |
