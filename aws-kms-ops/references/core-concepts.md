# KMS Core Concepts

AWS KMS (Key Management Service) architecture, components, and operational concepts.

## Service Overview

**AWS KMS** - Fully managed service for creating and controlling cryptographic keys used to encrypt data.

**Key Benefits:**
- Centralized key management
- Automatic key rotation
- Hardware Security Module (HSM) backed
- FIPS 140-2 Level 2 validated
- Integration with 100+ AWS services
- Fine-grained access control via IAM and key policies
- Audit logging via CloudTrail
- Multi-region key support

## Key Types

### Customer Managed Keys (CMK)
**Characteristics:**
- Created and managed by customer
- Full control over key lifecycle
- Custom key policies
- Enable/disable/schedule deletion
- Automatic or manual rotation
- Charged for key storage and usage

**Use Cases:**
- Application data encryption
- Customer data protection
- Cross-account access control
- Regulatory compliance requirements

### AWS Managed Keys
**Characteristics:**
- Created automatically by AWS services
- Cannot be directly managed by customer
- No separate charge for key storage
- Automatic annual rotation
- Service-specific (e.g., aws/s3, aws/rds)

**Services Using AWS Managed Keys:**
- S3 (SSE-S3 default encryption)
- EBS (default encryption)
- RDS (default encryption)
- SSM Parameter Store (SecureString)
- Many other AWS services

**Limitations:**
- Cannot view key policy
- Cannot disable/delete
- Cannot change rotation schedule
- Cannot use for direct encryption via API

### AWS Owned Keys
**Characteristics:**
- Fully owned by AWS
- No AWS account association
- Used for service-level encryption
- Not visible to customers
- No separate charges

**Use Cases:**
- Service-internal encryption
- Cross-service communication
- AWS operational data

## Key Specifications

### Symmetric Keys

#### SYMMETRIC_DEFAULT (AES-256-GCM)
**Characteristics:**
- 256-bit Advanced Encryption Standard
- Galois/Counter Mode (GCM)
- Authenticated encryption
- Same key for encrypt and decrypt
- Hardware-accelerated

**Operations:**
- Encrypt
- Decrypt
- GenerateDataKey
- GenerateRandom

**Use Cases:**
- Data encryption
- Envelope encryption
- Most common use case

**Performance:**
- API call limit: ~10,000 requests/second per key
- Automatic scaling
- No request throttling under normal usage

### Asymmetric Keys

#### RSA Keys
**Key Specs:**
- RSA_2048 (2,048 bits)
- RSA_3072 (3,072 bits)
- RSA_4096 (4,096 bits)

**Characteristics:**
- Public key for encryption
- Private key for decryption
- Digital signatures
- Slower than symmetric
- Larger key sizes

**Operations (ENCRYPT_DECRYPT):**
- Encrypt (with public key)
- Decrypt (with private key)

**Operations (SIGN_VERIFY):**
- Sign (with private key)
- Verify (with public key)

**Performance:**
- RSA_2048: ~500 ops/sec
- RSA_3072: ~100 ops/sec
- RSA_4096: ~50 ops/sec

**Use Cases:**
- Token signing
- Digital certificates
- Key exchange protocols

#### ECC Keys (Elliptic Curve)
**Key Specs:**
- ECC_NIST_P256 (secp256r1)
- ECC_NIST_P384 (secp384r1)
- ECC_SECG_P256K1 (secp256k1 - Bitcoin curve)

**Characteristics:**
- Smaller keys, same security
- Faster operations than RSA
- Digital signatures only
- Cannot encrypt

**Operations:**
- Sign
- Verify

**Performance:**
- ~1,000+ ops/sec
- Much faster than RSA

**Use Cases:**
- Cryptocurrency/blockchain
- IoT device authentication
- TLS certificates
- JWT signing

### SM2 Keys (China Regions)
**SM2**: Chinese national cryptographic algorithm
- Available in China (Beijing, Ningxia) regions
- Signature and encryption
- SM3 for hashing
- SM4 for symmetric encryption

## Key Origins

### AWS_KMS
**Characteristics:**
- Key material generated by AWS
- Stored in AWS HSMs
- Automatic key rotation
- Standard KMS operations

**Best for:**
- Most use cases
- Standard encryption
- When you don't need to import keys

### EXTERNAL
**Characteristics:**
- Key material imported by customer
- Must be 256-bit symmetric key
- Optional expiration
- Can delete key material, retain key
- No automatic rotation

**Import Process:**
1. Create key with EXTERNAL origin
2. Get public key for wrapping
3. Wrap key material with public key
4. Import wrapped key material
5. Set optional expiration

**Use Cases:**
- Compliance requires customer key generation
- Bring your own key (BYOK)
- Key escrow requirements
- Existing key migration

### AWS_CLOUDHSM
**Characteristics:**
- Key material in CloudHSM cluster
- Customer manages HSM
- Custom key store
- Higher performance

**Requirements:**
- CloudHSM cluster
- Custom key store
- IAM permissions for CloudHSM

**Use Cases:**
- FIPS 140-2 Level 3 requirements
- Dedicated HSM
- High-performance encryption

## Key Lifecycle

### States

#### Creating
- Initial state
- Brief duration
- Cannot use

#### Enabled
- **Normal active state**
- Full operations allowed
- Can encrypt/decrypt
- Can be disabled

#### Disabled
- **Temporarily inactive**
- Cannot encrypt new data
- Existing encrypted data still decryptable
- Can be re-enabled
- Monthly charge continues

#### PendingDeletion
- **Scheduled for deletion**
- Cannot use for any operation
- Can cancel during pending window
- Permanent deletion after window
- **Minimum 7 days, maximum 30 days**

#### PendingImport
- **Waiting for external key material**
- Import key material operation
- Cannot use until imported
- Can delete if not needed

#### Unavailable
- **Key material temporarily unavailable**
- Usually CloudHSM issue
- Automatic recovery
- May require manual intervention

### State Transitions

```
Creating → Enabled
         ↓
    Enabled ←→ Disabled
         ↓
    PendingDeletion → Deleted
         ↑
    Cancelled

Creating → PendingImport → Enabled (after import)
                              ↓
                         Deleted (material deleted)
```

## Key Policies

### Structure
Key policy is a resource-based IAM policy attached to key.

### Required Elements

#### Root User Permissions
```json
{
  "Sid": "Enable IAM User Permissions",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::{{account_id}}:root"
  },
  "Action": "kms:*",
  "Resource": "*"
}
```

**Purpose:**
- IAM policies can control access
- Prevents accidental lockout
- Account root always has access

### Common Policy Patterns

#### Allow Specific IAM Role
```json
{
  "Sid": "Allow Specific Role",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::{{account_id}}:role/{{role_name}}"
  },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey*",
    "kms:DescribeKey"
  ],
  "Resource": "*"
}
```

#### Allow AWS Service
```json
{
  "Sid": "Allow S3 Service",
  "Effect": "Allow",
  "Principal": {
    "Service": "s3.amazonaws.com"
  },
  "Action": [
    "kms:GenerateDataKey*",
    "kms:Decrypt"
  ],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:CallerAccount": "{{account_id}}",
      "kms:ViaService": "s3.{{region}}.amazonaws.com"
    }
  }
}
```

#### Cross-Account Access
```json
{
  "Sid": "Allow Cross-Account",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::{{other_account_id}}:root"
  },
  "Action": [
    "kms:Encrypt",
    "kms:Decrypt",
    "kms:GenerateDataKey*",
    "kms:DescribeKey"
  ],
  "Resource": "*"
}
```

### Policy Evaluation

**Effective Permissions = Key Policy AND IAM Policy**

Both must allow the action.

**Explicit Deny takes precedence over Allow.**

## Grants

### Purpose
Temporary, fine-grained permissions for key access.

### Characteristics
- Created by key owner/admin
- Specific to key
- Can include constraints
- Used by AWS services (e.g., S3, EBS)
- Can be retired or revoked

### Grant Components

#### Grantee Principal
- IAM user/role ARN
- AWS service principal
- Cannot be root account

#### Operations
- Encrypt
- Decrypt
- GenerateDataKey
- GenerateDataKeyWithoutPlaintext
- ReEncryptFrom
- ReEncryptTo
- Sign
- Verify
- GetPublicKey
- CreateGrant
- RetireGrant
- DescribeKey

#### Constraints
- EncryptionContextSubset: Required context
- EncryptionContextEquals: Exact context match

### Use Cases

#### S3 Server-Side Encryption
S3 creates grant to:
- Generate data key for PUT
- Decrypt data key for GET

#### Cross-Account Access
Without modifying key policy:
- Create grant for specific operations
- Set expiration (optional)
- Revoke when no longer needed

#### Temporary Access
- Grant for limited time
- Specific operations only
- Can be retired by grantee

## Encryption Context

### Purpose
Additional authenticated data for encryption.

### Characteristics
- Key-value pairs
- Stored with ciphertext
- Required for decryption
- Integrity verification
- Access control

### Benefits
- **Data integrity**: Context must match
- **Audit trail**: Context in CloudTrail logs
- **Access control**: Policy conditions
- **Key derivation**: Different context = different encryption

### Example
```
EncryptionContext: {
    "key": "value",
    "service": "myapp",
    "environment": "prod"
}
```

### Access Control with Context
```json
{
  "Sid": "Allow Only Specific Context",
  "Effect": "Allow",
  "Principal": {"AWS": "arn:aws:iam::{{account_id}}:role/{{role}}"},
  "Action": "kms:Decrypt",
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "kms:EncryptionContext:service": "myapp",
      "kms:EncryptionContext:environment": "prod"
    }
  }
}
```

## Key Rotation

### Automatic Rotation

#### Symmetric Keys
- **Annual rotation** (default)
- Automatic (configurable)
- Creates new key material
- Old material preserved
- Existing data remains readable
- New encryption uses new material

#### Rotation Process
```
1. Create new key material
2. Mark old material deprecated
3. New encrypt operations use new material
4. Decrypt uses appropriate material (automatic)
5. Old material kept indefinitely for decrypt
```

#### Asymmetric Keys
- **No automatic rotation**
- Must rotate manually
- Create new key
- Update aliases
- Retire old key

### Manual Rotation

#### For Symmetric Keys
```
1. Create new key
2. Create alias pointing to new key
3. Re-encrypt existing data (optional)
4. Schedule old key deletion
```

#### For Asymmetric Keys
```
1. Create new key pair
2. Update applications to use new key
3. Sign new data with new key
4. Maintain old key for verification
5. Schedule old key deletion after grace period
```

## Envelope Encryption

### Concept
- Data encrypted with data key (DEK)
- Data key encrypted with KMS key (KEK)
- Encrypted data key stored with ciphertext

### Benefits
- **Performance**: Local encryption with data key
- **Security**: Data key protected by KMS
- **Flexibility**: Can decrypt locally with decrypted key
- **Scalability**: No KMS API call per data operation

### Pattern
```
┌─────────────────────────────────────────────────────┐
│ Encrypted Data (Local AES)                          │
├─────────────────────────────────────────────────────┤
│ Encrypted Data Key (KMS Encrypted)                    │
└─────────────────────────────────────────────────────┘

To Encrypt:
1. Generate data key from KMS
2. Encrypt data locally with data key
3. Encrypt data key with KMS key
4. Store: [encrypted data] + [encrypted data key]
5. Discard plaintext data key

To Decrypt:
1. Extract encrypted data key
2. Decrypt data key with KMS
3. Decrypt data locally
4. Discard plaintext data key
```

## Multi-Region Keys

### Purpose
Same key material in multiple regions.

### Characteristics
- Primary key in one region
- Replica keys in other regions
- Same key ID across regions
- Same key material
- Independent key policies per region

### Use Cases
- Disaster recovery
- Cross-region data processing
- Reduced latency
- Compliance requirements

### Replication
```
Primary: us-east-1
    ↓
Replica: us-west-2
Replica: eu-west-1
```

### Operations
- Encrypt in any region
- Decrypt in any region
- Primary region for key management
- Can failover primary region

## Pricing Components

### Customer Managed Keys
- **$1.00/month** per key
- Prorated for partial months
- Charged while key exists
- Includes disabled keys
- Includes pending deletion keys

### API Requests
- **$0.03 per 10,000 requests**
- Encrypt, Decrypt, GenerateDataKey
- Sign, Verify, GetPublicKey
- DescribeKey (free)

### Data Transfer
- Standard AWS data transfer rates
- Cross-region for multi-region keys

### Free Tier
- **20,000 requests/month** free
- Applies to standard symmetric operations

## AIOps for KMS

### Automated Diagnosis (RCA)
| Error Pattern | Detection | Decision | Action |
|--------------|-----------|----------|--------|
| Decrypt fails | `describe-key` shows Disabled | [AUTO_HEAL] | `enable-key` |
| Key deletion scheduled | State=PendingDeletion | [AUTO_HEAL] | `cancel-key-deletion` |
| Access denied | IAM simulation fails | [MANUAL] | Review policies |
| API throttled | CloudWatch ThrottledRequests>1000 | [AI_ASSIST] | Implement caching |
| Rotation non-compliant | Symmetric key, rotation=false | [AUTO_HEAL] | `enable-key-rotation` |
| Unused key | No Decrypt in 90 days | [AI_ASSIST] | Consider deletion |

### Self-Healing Scenarios
```bash
# [AUTO_HEAL] Re-enable accidentally disabled key
aws kms enable-key --key-id {{key_id}}

# [AUTO_HEAL] Cancel unplanned deletion
aws kms cancel-key-deletion --key-id {{key_id}}

# [AUTO_HEAL] Enable rotation for production keys
aws kms enable-key-rotation --key-id {{key_id}}

# [AI_ASSIST] Identify unused keys for cost optimization
aws cloudtrail lookup-events --lookup-attributes AttributeKey=ResourceName,AttributeValue={{key_id}} --start-time $(date -d '-90 days' -u +%Y-%m-%dT00:00:00Z)
```

| CloudTrail Events | `lookup-events --lookup-attributes EventName=DisableKey` | Security auditing |

## Security Best Practices

### Key Management
- Use separate keys per application/environment
- Enable automatic rotation
- Use aliases for key references
- Document key purpose in description
- Tag keys for cost allocation

### Access Control
- Least privilege key policies
- Use encryption context for fine-grained control
- Audit CloudTrail logs
- Regular access reviews
- Remove unused grants

### Monitoring
- Monitor API call rates via CloudWatch
- Alert on DisableKey/ScheduleKeyDeletion via CloudTrail
- Track key state changes
- Audit key policy changes
- Monitor for unauthorized access

### Compliance
- Enable CloudTrail logging
- Use appropriate key specs for requirements
- Document key lifecycle
- Regular compliance reviews
- Proper key deletion procedures

## Common Integration Patterns

### S3 SSE-KMS
```
S3 Object Encryption:
1. S3 requests data key from KMS
2. KMS generates and encrypts data key
3. S3 encrypts object with data key
4. S3 stores encrypted object + encrypted key
```

### RDS Storage Encryption
```
RDS Encryption:
1. RDS uses KMS key for storage encryption
2. Automatic key rotation supported
3. Same key for backups/snapshots
4. Cross-region replication uses same key
```

### Lambda Environment Variables
```
Lambda Encryption:
1. KMS encrypts environment variable values
2. Lambda decrypts at runtime
3. Uses service-linked grant
4. Automatic key rotation supported
```

### EBS Volume Encryption
```
EBS Encryption:
1. EBS generates data key per volume
2. Data key encrypted with KMS key
3. EBS encrypts volume data with data key
4. Snapshots inherit encryption
```
