# Integration Setup (AWS Skills)

## Environment Setup

AWS CLI and boto3 SDK require a Python runtime. Use **`uv`** for local, isolated environment management.

### Install uv (One-time per machine)

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
# Or: brew install uv

# Windows (PowerShell)
powershell -c "irm https://astral.sh/uv/install.ps1 | iex"
```

### Bootstrap Python Environment (Idempotent)

```bash
uv venv --python 3.10

# Activate: macOS/Linux
source .venv/bin/activate
# Activate: Windows
# .venv\Scripts\activate

uv pip install awscli boto3
```

### Verify Installation

```bash
aws --version
python -c "import boto3; print('boto3 OK')"
```

## Credential Configuration

### Method A: IAM Role (AWS Runtime — Preferred)

When running on AWS compute (EC2, Lambda, ECS, EKS), attach an IAM Role. **No Access Key needed.**

```bash
# No env vars needed — boto3/CLI auto-discover instance metadata
aws sts get-caller-identity --output json
```

### Method B: AWS SSO / IAM Identity Center (Local Dev — Recommended)

AWS recommends SSO for human local development. No long-term Access Key.

```bash
# One-time setup
aws configure sso
# SSO start URL: https://your-org.awsapps.com/start
# SSO region: us-east-1
# Default client registration: None

# Login (creates temporary credentials, auto-refreshes)
aws sso login --profile my-sso-profile

# Use the profile
export AWS_PROFILE=my-sso-profile
aws sts get-caller-identity --output json
```

**~/.aws/config** (auto-generated by `aws configure sso`)
```ini
[profile my-sso-profile]
sso_start_url = https://your-org.awsapps.com/start
sso_region = us-east-1
sso_account_id = 123456789012
sso_role_name = AdministratorAccess
region = us-east-1
output = json
```

### Method C: STS AssumeRole (Cross-Account / Temporary)

Use temporary credentials for cross-account access or enhanced security.

```bash
# Assume a role
aws sts assume-role \
  --role-arn arn:aws:iam::TARGET_ACCOUNT:role/CrossAccountRole \
  --role-session-name skill-session \
  --duration-seconds 3600 \
  --output json

# Export the temporary credentials from response
export AWS_ACCESS_KEY_ID="..."
export AWS_SECRET_ACCESS_KEY="..."
export AWS_SESSION_TOKEN="..."
```

**~/.aws/config** (with source_profile for chained roles)
```ini
[profile cross-account]
role_arn = arn:aws:iam::TARGET_ACCOUNT:role/CrossAccountRole
source_profile = default
region = us-east-1
```

### Method D: Environment Variables (SDK / CI)

Use for CI/CD or when config files are unavailable. Prefer temporary credentials.

```bash
# Temporary credentials (recommended)
export AWS_ACCESS_KEY_ID="temporary_access_key"
export AWS_SECRET_ACCESS_KEY="temporary_secret_key"
export AWS_SESSION_TOKEN="temporary_session_token"
export AWS_DEFAULT_REGION="us-east-1"

# Or long-term keys (last resort — rotate every 90 days)
# export AWS_ACCESS_KEY_ID="your_access_key"
# export AWS_SECRET_ACCESS_KEY="your_secret_key"
```

### Method E: AWS CLI Config Files

**~/.aws/credentials**
```ini
[default]
aws_access_key_id = your_access_key
aws_secret_access_key = your_secret_key
```

**~/.aws/config**
```ini
[default]
region = us-east-1
output = json
```

## Credential Priority Order

AWS SDK/CLI auto-discovery chain (highest → lowest):

| Priority | Source | Used By | Notes |
|----------|--------|---------|-------|
| 1 | `AWS_PROFILE` env var | CLI + SDK | Points to named profile in `~/.aws/config` |
| 2 | Explicit env vars (`AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY`) | CLI + SDK | Includes `AWS_SESSION_TOKEN` for temporary creds |
| 3 | `~/.aws/credentials` | CLI + SDK | Named profiles supported |
| 4 | `~/.aws/config` | CLI + SDK | SSO / AssumeRole source_profile |
| 5 | IAM Role (instance metadata) | CLI + SDK | EC2/Lambda/ECS/EKS — automatic, no config |

**Note**: If `AWS_PROFILE` is set, SDK/CLI will use that profile's configuration (including SSO or chained AssumeRole) and skip other credential sources.

## Verify Credentials

```bash
aws sts get-caller-identity --output json
```

Expected output:
```json
{
  "UserId": "AIDAI...",
  "Account": "123456789012",
  "Arn": "arn:aws:iam::123456789012:user/username"
}
```

Agent MUST log each credential resolution step:

```

[OK]   AWS_DEFAULT_REGION=us-east-1 (from .env)
[OK]   AWS_ACCESS_KEY_ID=AKIA... (from .env)
[OK]   AWS_SECRET_ACCESS_KEY=**** (from .env, masked)
[OK]   AWS_SESSION_TOKEN=**** (from .env, masked, optional)

```

Or with profile:
```

[OK]   AWS_PROFILE=sso-admin (from .env)
```

Or with IAM Role:
```

[OK]   IAM Role detected: arn:aws:iam::123456789012:role/MyRole
[INFO] Using IAM Role (no keys needed)
```

## Error Messages (Precise & Actionable)

### No Credentials Found
```
[FAIL] AWS credentials not found.

Please configure ONE of the following:

1. Access Keys (local dev):
   Copy .env.example → .env, then fill in:
   AWS_ACCESS_KEY_ID=your_key
   AWS_SECRET_ACCESS_KEY=your_secret

2. Named Profile (SSO / multi-account):
   AWS_PROFILE=your-profile-name
   Ensure ~/.aws/config has [profile your-profile-name]

3. IAM Role (AWS compute):
   No configuration needed on EC2/Lambda/ECS/EKS.
   Attach an IAM Role to the resource.

Diagnose: aws sts get-caller-identity --output json
```

### Invalid Credentials
```
[FAIL] AWS credential verification failed.

AWS Error: InvalidClientTokenId — The security token included in the request is invalid.

Common causes:
  - AWS_ACCESS_KEY_ID is incorrect or deleted
  - AWS_SECRET_ACCESS_KEY has a typo
  - AWS_SESSION_TOKEN expired (temporary credentials)

Actions:
  1. Verify keys in AWS Console → IAM → Security credentials
  2. If using temporary credentials, refresh the session token
  3. Run 'aws sts get-caller-identity' manually to confirm
```

### Missing Region
```
[WARN] AWS_DEFAULT_REGION not set.

Defaulting to us-east-1.
To override, set AWS_DEFAULT_REGION in .env or shell:
  export AWS_DEFAULT_REGION=ap-northeast-1

Available regions: https://docs.aws.amazon.com/general/latest/gr/rande.html
```

```ini

# JD Cloud - use JDC_* prefix

```

## Safety Rules

- **NEVER** commit `.env` files (already in `.gitignore`)
- **NEVER** write credentials into Skill documents
- Generated Skills use `{{env.*}}` placeholders only

## `.env` to Skill Config Injection

Skill `assets/example-config.yaml` files use `{{env.*}}` placeholders for environment-level values (region, account ID) and `{{user.*}}` for resource-specific values. Agent behavior:

1. Before loading any `example-config.yaml`, check for `.env` in project root
2. Load `.env` variables (shell env overrides `.env` file)
3. Substitute all `{{env.VAR_NAME}}` placeholders with loaded values
4. Collect `{{user.*}}` values from user input
5. Use the rendered configuration for CLI/SDK commands

| Placeholder | `.env` Variable | Used In |
|-------------|-----------------|---------|
| `{{env.AWS_DEFAULT_REGION}}` | `AWS_DEFAULT_REGION` | Region fields, ARNs |
| `{{env.AWS_ACCOUNT_ID}}` | `AWS_ACCOUNT_ID` | ARN account segment |
| `{{env.AWS_DEFAULT_VPC_ID}}` | `AWS_DEFAULT_VPC_ID` | VPC references |
| `{{env.AWS_DEFAULT_SUBNET_ID}}` | `AWS_DEFAULT_SUBNET_ID` | Subnet references |

**Priority order** (same as credential loading):
1. Shell environment variables (highest)
2. `.env` file values
3. Default fallback values (lowest)
